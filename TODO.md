## 트랙 1 — 방법론 (Protocol / Methodology)

추출 방법 11개 문서가 현재 **설계 설명** 수준이다. 연구자가 문서만 보고 독립 재현하려면 **절차서** 수준이 돼야 한다.

### T1-1. 퍼실리테이터 역할 정의 ✅ 확정
대부분의 메서드는 고정 프로토콜이라 퍼실리테이터 불필요. LLM이 퍼실리테이터인 것은 직전 답변을 보고 다음 질문을 동적으로 결정해야 하는 2개뿐.

**LLM 퍼실리테이터 필요 (2개)**
- 래더링 — "왜 그게 중요해요?"를 답변 맥락에 따라 반복 프로빙
- CCRT — 관계 에피소드를 끌어내고 맥락에 맞는 후속 질문

**고정 프로토콜 · 자기 진행 (9개)** — 퍼실리테이터 불필요
- 삼항 도출 (조합 선택은 dispersion·saliency 알고리즘), 두려운 자기, 대표 인생 장면, 가치 할당, 생성 은유·SCT, 판단 스타일, 자기-타자, CIT, ESM

남은 작업: 래더링·CCRT 문서에 LLM 퍼실리테이터 동작 방식을 프로토콜 수준으로 명세.

### T1-2. 중단 기준 명시 ✅ 확정

**래더링** — 최대 4단계. 다음 중 하나 도달 시 종료:
- 종착 가치 감지: 참가자가 "그냥 중요해서요" / "이게 내 삶의 의미니까요" 류 반응 → 더 파면 동어반복
- 4단계 도달 → 강제 종료 (4 초과는 추상화 과잉, 검증 불가 영역)
- 반복 감지: 이전 답변과 의미 동일 응답 나오면 즉시 종료

**삼항** — 다음 중 먼저 도달하는 기준으로 종료:
- 포화(saturation): 3라운드 연속 새로운 구성개념이 나오지 않음
- 상한: 구성개념 12개 도달
- 실용 하한: 최소 6개는 있어야 PCA 의미 있음

**CCRT** — 에피소드 기반:
- 최소 3개 (Luborsky 원 연구 기준 — 3개부터 W·RO·RS 패턴 식별 가능)
- 권장 5개 (안정적 패턴 확인용)
- 상한 없음 — 많을수록 패턴 신뢰도 높아짐

**SCT·생성 은유** — 문항 수 고정. 항상 같은 문항 세트. 가변 불가 (오염 방지).

**ESM** — 세션 단위 아님. 하루 N회 × Y일 지속. N·Y는 T2에서 결정.

### T1-3. 수치 계산 절차 ✅ 완료
→ [계산 절차](<계산 절차.md>) 신설. 삼항 그리드 분석(dispersion·saliency·PCA·딜레마)과 검증 지표(정규화 정확도·Brier·n-AFC) 전부 수식 + Python 코드 포함.

### T1-4. 메서드 간 의존 순서 공식화 ✅ 확정

**고정 의존 (순서 변경 불가)**
- 삼항 도출 → 래더링 (래더링은 삼항 construct axis를 입력으로 받음)

**권장 순서 (낮은 피로 → 높은 피로)**
1. 생성 은유·SCT (창의적, 워밍업)
2. 가치 할당 (구체 시나리오, 인지 부하 중간)
3. 두려운 자기 (자기 직면, 의존성 없음)
4. 삼항 도출 (집중력 필요, 래더링 선행)
5. 래더링 (삼항 필수 선행)
6. CCRT (관계 기억 소환, 감정 부하 높음)
7. 대표 인생 장면 (서사, 중간)
8. CIT (구체 사건 분석, 중간)
9. 판단 스타일 (인지적, 비교적 가벼움)
10. 자기-타자 (메타 지각, 언제든 가능)
11. ESM (진행형, 단일 세션 아님)

**순서 독립** — 삼항·래더링 묶음 외 나머지 9개는 순서 자유. 위 권장 순서는 피로도 관리를 위한 권고일 뿐.

**도메인별 최소 메서드 조합** — 고정 수치 없음. 검증 관문 통과(정규화 정확도 > 콜드 베이스라인 + 임계치)를 해금 기준으로 삼는다. 임계치는 ablation 학습곡선으로 결정. 출시 전 실용 기준: 해당 도메인에서 삼항+래더링 최소 1세트 완료.

### T1-5. 에지케이스 (방법론 레벨)
- 참가자가 단답(한 단어)만 답할 때
- 세션 중간에 멈춘 경우 데이터를 쓸 수 있는가
- 한 도메인에 에피소드가 1개밖에 없을 때 CCRT를 쓸 수 있는가

## 트랙 2 — 서비스 스펙 (Service Spec)

[MVP 기획](<MVP 기획.md>)이 기반이다. 거기서 빠진 레이어들.

### T2-1. 화면 흐름 상세 (Screen Flow)
MVP 기획 §4에 화면별 설계가 있지만 **화면 간 전환 로직**이 없다.
- 각 화면의 진입 조건, 이탈 조건
- 분기 (도메인 해금 여부, 토글 상태, 검증 통과 여부)에 따른 화면 변화
- 모바일/웹 레이아웃 구분이 필요한지

### T2-2. 세션 상태 머신
추출 세션 하나가 가질 수 있는 상태와 전환이 정의돼야 한다.
- 상태: not_started / in_progress / paused / completed / abandoned
- 중간 저장 시점
- 재개 시 어느 시점부터 이어받는가
- 세션과 데이터 버전 관리

### T2-3. 데이터 모델
raw_store.yaml은 로컬 포맷이다. 서비스 레벨 데이터 모델이 필요하다.
- User, Session, Entry, ConsentToggle, VerificationRound 등 엔티티
- 친구 연결 구조 (누가 누구의 데이터를 어떤 용도로 볼 수 있는가)
- contaminated 플래그 처리

### T2-4. LLM 연동 스펙 ✅ 확정

**분석 모드 — 2-step 구조**
1. 사용자 텍스트 입력 → LLM-A가 요청 의도 + 데이터 스키마를 보고 "분석에 적합한 프롬프트" 생성 (실제 데이터 미포함, 스키마만)
2. 생성된 프롬프트 + 실제 raw_store 데이터 → LLM-B → 최종 출력
→ 사용자 입력이 분석 방향을 결정하지만, 프롬프트 설계는 LLM이 자동화

**시뮬레이션 모드**
- 에이전트 초기화: 시스템 프롬프트(경량) + 해당 참가자 raw_store 전체 주입
- 턴 구조: 기존 멀티에이전트 시뮬레이터 활용 (직접 구현 X)
- 관찰자 UI: 대화 내용 + 행동 결과 텍스트 출력만. 환경 그래픽화는 나중

**검증 모드**
- 봉인 예측: raw_store + 예측 대상 질문 → LLM → 예측값 제출 (사용자에게 봉인 후 공개)
- 라인업 미끼: 다른 참가자 익명 응답들 풀에서 n-1개 선택

**컨텍스트 길이**: 문제 없음. 현재 모델 100만 토큰 전후. 11개 메서드 전체 주입 가능.

### T2-5. 검증 관문 구현
MVP 기획 §5에 개념은 있는데 구현 레벨이 없다.
- 봉인 예측을 어떻게 봉인하는가 (사용자가 못 보도록)
- 라인업 미끼(n-AFC)를 어떻게 구성하는가 (타인 데이터 익명화 풀)
- 원장(test ledger) 스키마
- 검증 통과 판정 기준 (수치)

### T2-6. 에지케이스 (서비스 레벨)
- 친구가 동의를 중간에 철회하면 이미 돌아간 분석 결과는 어떻게 되나
- 검증 라운드 도중 앱을 끄면
- 같은 데이터로 오염 전/후 검증이 섞이는 경우 분리 처리
- 미성년자 / 위기 신호 감지 분기

### T2-7. UI/UX 상세 ✅ 확정 (self-explanatory 처리)

대부분 구현 시점에 자연스럽게 결정되는 것들. 지금 스펙화 필요 없음.

- **검사 진행 화면**: self-explanatory. 문항 → 입력 → 다음. 메서드별로 자명하게 그려짐.
- **래더링 UI**: 텍스트 대립축 + 올라가는 사슬. 머메이드 다이어그램 스타일. 예쁜 건 나중.
- **가치 할당 UI**: 버튼으로 점수 배분, 합계 100 강제. 명확함.
- **시뮬레이션 관찰 UI**: 환경 그래픽화 나중. 지금은 대화 내용 + 행동 결과 텍스트 출력만.
- **검증 봉인/공개**: self-explanatory. 봉인 → 사용자 실제 응답 → 공개·대조.
