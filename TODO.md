## 트랙 1 — 방법론 (Protocol / Methodology)

추출 방법 11개 문서가 현재 **설계 설명** 수준이다. 연구자가 문서만 보고 독립 재현하려면 **절차서** 수준이 돼야 한다.

### T1-1. 퍼실리테이터 역할 정의 ✅ 확정
대부분의 메서드는 고정 프로토콜이라 퍼실리테이터 불필요. LLM이 퍼실리테이터인 것은 직전 답변을 보고 다음 질문을 동적으로 결정해야 하는 1개뿐.

**LLM 퍼실리테이터 필요 (1개)**
- 래더링 — "왜 그게 중요해요?"를 답변 맥락에 따라 반복 프로빙

**고정 프로토콜 · 자기 진행 (10개)** — 퍼실리테이터 불필요
- 삼항 도출 (조합 선택은 dispersion·saliency 알고리즘), 두려운 자기, 대표 인생 장면, 가치 할당, 생성 은유·SCT, 판단 스타일, 자기-타자, CIT, ESM, CCRT (고정된 개방형 질문에 스스로 답함)

남은 작업: 래더링 문서에 LLM 퍼실리테이터 동작 방식을 프로토콜 수준으로 명세.

### T1-2. 중단 기준 명시 ✅ 확정

**래더링** — 최대 4단계. 다음 중 하나 도달 시 종료:
- 종착 가치 감지: 참가자가 "그냥 중요해서요" / "이게 내 삶의 의미니까요" 류 반응 → 더 파면 동어반복
- 4단계 도달 → 강제 종료 (4 초과는 추상화 과잉, 검증 불가 영역)
- 반복 감지: 이전 답변과 의미 동일 응답 나오면 즉시 종료

**삼항** — 다음 중 먼저 도달하는 기준으로 종료:
- 포화(saturation): 3라운드 연속 새로운 구성개념이 나오지 않음
- 상한: 구성개념 12개 도달
- 실용 하한: 최소 6개는 있어야 PCA 의미 있음

**CCRT** — 에피소드 기반:
- 최소 3개 (임상에서는 Luborsky(1998) 기준 연구용 최소 10개를 권장하나, MVP 피로도와 초기 패턴 감지를 위해 실용 하한을 3개로 타협. 3개부터 초기 W·RO·RS 패턴 식별 가능)
- 권장 5개 (안정적 패턴 확인용)
- 상한 없음 — 많을수록 패턴 신뢰도 높아짐

**SCT·생성 은유** — 문항 수 고정. 항상 같은 문항 세트. 가변 불가 (오염 방지).

**ESM** — 세션 단위 아님. 하루 N회 × Y일 지속. N·Y는 T2에서 결정.

### T1-3. 수치 계산 절차 ✅ 완료
→ [계산 절차](<계산 절차.md>) 신설. 삼항 그리드 분석(dispersion·saliency·PCA·딜레마)과 검증 지표(정규화 정확도·Brier·n-AFC) 전부 수식 + Python 코드 포함.

### T1-4. 메서드 간 의존 순서 공식화 ✅ 확정

**고정 의존 (순서 변경 불가)**
- 삼항 도출 → 래더링 (래더링은 삼항 construct axis를 입력으로 받음)

**권장 순서 (낮은 피로 → 높은 피로)**
1. 생성 은유·SCT (창의적, 워밍업)
2. 가치 할당 (구체 시나리오, 인지 부하 중간)
3. 두려운 자기 (자기 직면, 의존성 없음)
4. 삼항 도출 (집중력 필요, 래더링 선행)
5. 래더링 (삼항 필수 선행)
6. CCRT (관계 기억 소환, 감정 부하 높음)
7. 대표 인생 장면 (서사, 중간)
8. CIT (구체 사건 분석, 중간)
9. 판단 스타일 (인지적, 비교적 가벼움)
10. 자기-타자 (메타 지각, 언제든 가능)
11. ESM (진행형, 단일 세션 아님)

**순서 독립** — 삼항·래더링 묶음 외 나머지 9개는 순서 자유. 위 권장 순서는 피로도 관리를 위한 권고일 뿐.

**도메인별 최소 메서드 조합** — 고정 수치 없음. 검증 관문 통과(정규화 정확도 > 콜드 베이스라인 + 임계치)를 해금 기준으로 삼는다. 임계치는 ablation 학습곡선으로 결정. 출시 전 실용 기준: 해당 도메인에서 삼항+래더링 최소 1세트 완료.

### T1-5. 응답 품질 처리 ✅ 확정

**원칙: 이상한 답변도 데이터다.**
단답, 회피, 농담식 응답, 맥락 없는 한 단어 — 이것도 그 사람의 스타일이다. 임상심리학에서 TAT·로르샤흐·개방형 면접 모두 짧거나 비전형적인 응답을 해석 대상으로 다룬다. 거부·침묵도 반응이다. 거르지 않고 그대로 저장하되, 해석 단계에서 LLM이 응답 스타일 자체를 데이터로 읽는다.

**두 가지만 필터링**
1. **형식 오류** (알고리즘/LLM): 필드 미입력, 타입 오류, YAML 파싱 불가 등 — 재입력 요청
2. **완전한 무응답** (알고리즘): 공백·null만 있는 경우 — 선택적으로 재시도 유도 (강제 아님)

**형식 가이드라인 (참가자에게 제시)**
- 1인칭으로 쓸 것 ("나는 ~")
- 최소 한 문장 이상
- 그 순간 실제로 든 생각·감각·행동 위주
- 정답 없음 — 떠오르는 그대로

**세션 중단**: 그때까지 저장된 항목 그대로 사용. 미완성 세션도 유효한 부분 데이터.

**CCRT 1개 에피소드**: 패턴 주장 불가. 저장은 하되 분석 시 "단일 에피소드, 패턴 미확인" 플래그.

### T2-4. LLM 연동 스펙 ✅ 확정

**분석 모드 — 2-step 구조**
1. 사용자 텍스트 입력 → LLM-A가 요청 의도 + 데이터 스키마를 보고 "분석에 적합한 프롬프트" 생성 (실제 데이터 미포함, 스키마만)
2. 생성된 프롬프트 + 실제 raw_store 데이터 → LLM-B → 최종 출력
→ 사용자 입력이 분석 방향을 결정하지만, 프롬프트 설계는 LLM이 자동화

**시뮬레이션 모드**
- 에이전트 초기화: 시스템 프롬프트(경량) + 해당 참가자 raw_store 전체 주입
- 턴 구조: 기존 멀티에이전트 시뮬레이터 활용 (직접 구현 X)
- 관찰자 UI: 대화 내용 + 행동 결과 텍스트 출력만. 환경 그래픽화는 나중

**검증 모드**
- 봉인 예측: raw_store + 예측 대상 질문 → LLM → 예측값 제출 (사용자에게 봉인 후 공개)
- 라인업 미끼: 다른 참가자 익명 응답들 풀에서 n-1개 선택

**컨텍스트 길이**: 문제 없음. 현재 모델 100만 토큰 전후. 11개 메서드 전체 주입 가능.

### T2-5. 검증 관문 구현 ✅ 완료

**봉인 예측**: 사용자 응답 전에 LLM이 먼저 예측값 생성 → 저장(봉인). 사용자가 실제 응답 완료 후 공개·대조. 순서가 핵심 — 뒤집히면 앵커링으로 데이터 오염.

**라인업 미끼**: 참가자 풀이 일정 수 이상일 때만 활성. 미만이면 A2 스킵, A1·B·C만으로 검증. (목격자 증언 심리학의 6-pack 라인업 표준에 따라, 타겟 외에 최소 5개의 미끼(foil)가 필요. 따라서 데이터베이스 내 유효 참가자가 **최소 6명 이상**일 때부터 활성)

**원장 스키마 (잠정)**
```
verification_ledger:
  - round_id
    session_id
    method: A1 | A2 | B1 | B2 | ...
    question
    llm_prediction
    actual_answer
    match: true | false | partial
    confidence_forecast: float  # Brier용
    contaminated: false         # 활용 이후 재진입이면 true
    ts
```

**검증 통과 판정 기준**: soul.md 베이스라인 대비 측정.
- 콜드 베이스라인: LLM에게 soul.md (자기기술 프롬프트)만 주고 예측
- 우리 시스템: raw_store 전체 주입 후 예측
- 우리가 soul.md 베이스라인을 초과해야 해금 — 이게 A4 증분 타당도의 실제 구현
- 수치 임계치는 ablation 후 결정. placeholder: soul.md 정확도 + 10%p

## 트랙 7 — 엔지니어링 관점의 구현 블로커 및 누락 사항 (Engineering Implementation Blockers)

문서를 바탕으로 실제 프로덕트(웹/앱) 코드를 작성하려 할 때, 현재 기획만으로는 개발자가 단독으로 결정을 내리기 어렵거나 구조적으로 비어있는(Missing) 영역들입니다.

### E-1. LLM 오케스트레이션 및 데이터 파이프라인 (LLM Orchestration)
- **프롬프트 버전 관리 (Prompt Versioning):** 프롬프트 내용이 수정되면 과거에 해당 프롬프트로 진행했던 "검증 모드" 결과나 "추출" 데이터의 정합성이 무너집니다. `raw_store`나 검증 원장에 프롬프트 해시값/버전(예: `v1.2`)을 함께 로깅하는 버저닝 전략이 문서에 없습니다.
- **비동기 스트리밍(SSE/WebSockets) 스펙 미비:** 래더링(동적 꼬리질문)이나 챗봇 모드 등은 사용자 경험 상 타이핑 효과(스트리밍)가 필수입니다. 단순 REST API(POST)가 아닌 스트리밍 응답 처리에 대한 백엔드-프론트엔드 간 통신 아키텍처 명세가 빠져 있습니다.

### E-2. 상태 머신과 게이미피케이션 로직 (State & Progression)
- **도메인 경험치(XP) 및 검사 트리 구조 기획:** 사용자에게 RPG 같은 달성감을 주기 위해 아래의 경험치 맵을 제안합니다.
    - `일/관계/자기` 도메인별 게이지(100 XP 만점)를 개별 운영합니다.
    - **XP 보상 테이블:** 삼항 도출(+40 XP), 래더링(+30 XP), 가치 할당(+30 XP), 핵심 갈등 도식(+20 XP / 에피소드당)
    - **해금 트리거:** 특정 도메인이 100 XP 도달 시 해당 도메인의 검증 관문이 자동으로 열리도록 Rule 엔진을 작성합니다.
    - **DAG 추천 로직:** 현재 선택된 도메인에서 "가장 높은 XP를 주면서, 선행 조건(예: 삼항 도출 선행)이 충족된, 아직 진행하지 않은 검사"를 대시보드 최상단에 추천 노드로 띄워줍니다.
- **백엔드 기반 세션(Session) 복구 로직 제안:** 사용자가 호흡이 긴 검사 도중 이탈했을 때를 대비한 자동 임시저장(Auto-save) 명세입니다.
    - **클라이언트 측:** 사용자가 폼에 입력할 때마다 브라우저 `LocalStorage`에 실시간 캐싱합니다. (타이핑 유실 방지)
    - **서버 측:** 마지막 입력 후 3초가 지나면(Debounce) 백엔드 `/api/extract/draft` 엔드포인트로 JSON payload를 PUT 요청합니다. 이때 DB에는 `status: "draft"`로 저장됩니다.
    - **복구 UX:** 재진입 시 백엔드에 "draft" 상태의 세션이 있으면, "진행 중인 검사가 있습니다. 이어서 하시겠습니까?"라는 바텀 시트를 띄우고 수락 시 해당 JSON 데이터를 Form에 채워 넣습니다. 완료 시에만 `status: "completed"`로 변경하여 `raw_store` 유효 데이터로 편입시킵니다.

### E-3. 다중 에이전트 시뮬레이션의 엔지니어링 에지 케이스 (Multi-Agent Edge Cases)
- **LangGraph 기반 데드락(무한 루프) 방지 설계:** Agent A와 Agent B가 서로 "네 생각은 어때?" 하고 끝없이 대화하는 루프를 막기 위한 제안입니다.
    - **하드 턴 리밋 (Hard Turn Limit):** LangGraph의 `recursion_limit`을 10~15 턴 내외로 강제하여, 시나리오가 길어지면 그래프 실행이 자동 중단되도록 설계합니다.
    - **중재자 노드 (Facilitator Node):** A와 B 사이에 판사 역할을 하는 LLM 라우터 노드를 하나 추가합니다. 이 노드는 대화의 교착 상태(Stalemate)를 감지하면 "이제 결론을 내려보세요" 혹은 "새로운 상황이 주어졌습니다"라는 프롬프트를 주입하여 대화 양상을 강제로 전환시킵니다.
- **LangGraph 기반 동시성(Concurrency) 및 상태 동기화 제안:**
    - 두 에이전트가 동시에 생각(추론)하고 말하게 하는 것은 챗봇 환경에 부적합합니다. (레이스 컨디션 발생)
    - LangGraph의 순차적(Sequential) 그래프 플로우를 채택하여, `Node_A_Turn` → `Update_Global_State` → `Node_B_Turn` → `Update_Global_State` 구조로 상태 동기화를 단일 스레드처럼 처리합니다.
    - **체크포인팅 (Checkpointing):** LangGraph 내장 SQLite/Postgres 기반 Thread Checkpointer를 활용합니다. 시뮬레이션 중단/에러 시에도 마지막 턴의 글로벌 상태(State)가 영구 저장되므로, 사용자는 언제든 마지막 턴에서 이어서 시뮬레이션을 관전할 수 있습니다.

### E-4. 검증 관문(Validation Gate)의 엄밀성
- **프로브(Probe) 마스킹 알고리즘의 한계:** "해당 도메인의 데이터를 마스킹한다"고 되어 있으나, 자연어로 작성된 `raw_store`에서 특정 정보만 감쪽같이 도려내는 것은 매우 어렵습니다 (Data Leakage 우려). 정규식/Rule-based로 가릴 것인지, 별도의 LLM 전처리기를 두어 Masking할 것인지 기술적 명세가 명확하지 않습니다.
- **LLM-as-a-judge의 신뢰성 보정:** 검증 점수 평가를 LLM에게 맡기는데, 동일한 입력에도 Temperature나 모델 업데이트에 따라 점수가 달라질 수 있습니다. Logprobs 기반 확률 추출을 쓸 것인지, 다수결(Ensemble)을 할 것인지 채점의 엔지니어링 보정 장치(Determinism 확보)가 필요합니다.

### E-5. 프론트엔드 UI 컴포넌트 복잡도 (Complex UI State Management)
- **복합 UI 상호작용 명세 부족:** 단순 Text Form이 아니라 "단어 3개를 주고 2개 vs 1개로 그룹핑(삼항 도출)" 하거나 "나무뿌리처럼 이어지는 사슬(래더링)"을 그리는 UI는 프론트엔드 상태 관리가 매우 복잡합니다. 드래그 앤 드롭, 노드 간 연결, 반응형 렌더링 등 핵심 컴포넌트의 Interaction 명세가 부족하여 현재 문서의 "Self-explanatory(자명함)"라는 단어로 치부하기엔 개발 리스크(일정 지연)가 큽니다.
