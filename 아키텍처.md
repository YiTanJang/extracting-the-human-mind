# App Blueprint: 심리 분석 서비스 아키텍처 명세서

> **3줄 요약:**
> - 이 프로젝트가 실제 앱이나 웹 서비스로 만들어질 때 어떤 구조로 동작할지 보여주는 시스템 설계도이다.
> - 데이터를 뽑아내고(추출), 맞는지 확인하고(검증), 이를 바탕으로 챗봇 상담이나 시뮬레이션을 하는(활용) 3단계 구조를 정의한다.
> - 사용자의 개인정보를 보호하면서 데이터베이스에 어떻게 안전하게 저장할지 명시한다.
>
> **To AI Coding Agents:** 
> 이 문서는 기획에 정의된 사용자 여정을 실제 소프트웨어로 구현하기 위한 **엔지니어링 시스템 설계도이다.** 기획적인 의도보다는 **Information Architecture (IA), 시스템 모듈 구성, 데이터 파이프라인, RDBMS 스키마** 구현에 집중한다.

이 애플리케이션의 본질은 일반적인 소셜 앱이 아니다. **"추출 ➔ 검증 ➔ 활용"** 이라는 엄격한 단방향 파이프라인을 코드로 구현하는 것이 핵심이다. 친구 추가 등의 소셜 기능은 단지 이 파이프라인의 '활용' 단계에서 데이터 소스를 주입하기 위한 선택적 확장 레이어일 뿐이다.

## 1. Information Architecture (IA)

웹/앱의 라우팅 및 폴더 구조는 아래의 IA를 따른다. 사용자의 데이터 상태(해금 상태)에 따라 접근 권한이 통제되어야 한다.

```text
/ (Root)
├── /onboarding             # 이름 기입, 고유 닉네임 발급, 실험 주의서 동의
├── /auth                   # 초기 패스코드/초대 토큰 기반 인증 레이어
├── /hub                    # 검사 허브. 11가지 추출 모듈 렌더링.
│   ├── /hub/ccrt
│   ├── /hub/laddering
│   └── ... (진행률 UI 포함)
├── /validation             # 검증 관문. 특정 도메인 임계치 도달 시 활성화.
│    └── /validation/[test_id] # Hold-out 예측 테스트 실행 뷰
├── /modes                  # 활용 모드 허브. 검증 통과 시 해금.
│   ├── /modes/chat         # 대화/상담 모드 UI
│   ├── /modes/profiler     # 분석 모드 UI
│   └── /modes/sandbox      # 다중 에이전트 시뮬레이션 UI
└── /social                 # 타인 데이터 주입 레이어. 익명 피드백 초대 링크 발급 및 연동.
```

## 2. Core System Modules (시스템 플로우)

시스템은 3개의 독립적인 핵심 엔진으로 구성된다. 각 엔진은 앞선 엔진의 결과물만을 인풋으로 받는다.

### Module 1: Extraction Engine (추출 엔진)
- **Trigger:** 사용자가 `/hub`에서 특정 검사를 시작한다.
- **Logic:** 검사의 성격에 따라 두 가지 파이프라인으로 나뉜다.
    1. **정적 검사:** LLM이 개입하지 **않는다**. 앱은 정적인 위키 질문을 화면에 띄워주고, 사용자는 빈 텍스트 박스에 답을 적어 DB로 직행시킨다.
    2. **동적 검사:** 지속적인 꼬리질문이 필요하므로 LLM(챗봇)이 보조로 투입된다. 단, 시스템 프롬프트를 통해 **"절대 사용자의 의도를 지레짐작해 요약/라벨링하지 말고 오직 꼬리질문만 할 것을"** 강제한다.
- **데이터 적재 원칙:** 정적/동적 방식 모두 최종적으로 사용자가 입력한 텍스트는 **절대 요약되거나 환언되지 않고**, 날것 그대로 구조화되어 DB(`raw_store`)에 적재된다.
- **전역 가명화:** 프라이버시 보호를 위해 DB 저장 및 외부 LLM API 전송 직전에 **정규표현식 및 NER 기반의 가명화 미들웨어를** 반드시 거친다. 
    - **핵심 원칙:** 문장의 어투나 구조를 변형할 위험을 막기 위해, **문장의 다른 어떤 글자도 건드리지 않고 오직 실명만 `[Person_A]`, `[Company_B]` 등으로 치환하는** 결정론적 필터링을 강제한다.
- **LLM 하이퍼파라미터:** 환각을 막고 일관된 분석을 위해 추출 봇 및 검증 채점기는 **Temperature = 0.0 ~ 0.2로** 강제한다.
- **JSON Schema:** 프론트엔드 Form에서 DB로 넘어가는 데이터 규격이다.
    ```python
    class CCRTEntry(BaseModel):
        id: str
        method: Literal["ccrt"]
        object_tag: str
        raw_wish: str
        raw_response_other: str
        raw_response_self: str
        raw_recurrence: str
    ```

### Module 2: Validation Gatekeeper (검증 엔진) - 시스템의 코어
- **Trigger:** 특정 도메인의 데이터가 임계치에 도달한다.
- **Logic:**
    1. 서버 백엔드가 DB에서 해당 유저의 원본 데이터를 조회하되, **테스트용 타겟 데이터(Probe)를 마스킹 처리**한다.
    2. 마스킹하고 남은 데이터만 LLM에 주입하여 타겟 질문에 대한 응답을 예측시킨다.
    3. 프론트엔드 UI를 통해 사용자에게 실제 타겟 질문을 던져 **실제 정답을** 수집한다.
    4. **채점 알고리즘:** 물리적으로 분리된 평가용 STS 모델을 호출하여 예측 텍스트와 실제 정답 간의 논리적 일치도를 스칼라 값으로 출력받고 `test_ledger`에 기록한다.
    5. 최소 배터리를 오염 없이 적재 완료 시 해당 도메인의 `is_unlocked` 상태를 True로 업데이트한다. **정확도 점수가 낮아도 통과하며,** 낮은 점수도 방법론의 한계를 보여주는 데이터로 저장된다.

### Module 3: Utilization Engine (활용 엔진)
- **Trigger:** 사용자가 `/modes`에서 특정 모드를 실행한다.
- **Logic:**
    1. `is_unlocked`가 True인 데이터만 조회한다.
    2. **데이터 소스 결합:** 시뮬레이션에 타인을 포함했다면 내 배열과 친구의 배열을 함께 가져온다.
    3. **멀티 에이전트 아키텍처:** 다중 에이전트 시뮬레이션의 경우 토큰 오버플로우 및 자아 혼란을 방지하기 위해 **독립된 LLM 인스턴스를 각각 메모리에 띄우고**, 중앙 통제기(Environment State Machine / 판사 노드) 계층을 씌운 구조로 구현해야 한다. 
        - 에이전트들의 하이퍼파라미터는 자연스러운 롤플레잉 대화를 유도하기 위해 **Temperature = 0.7을** 기본값으로 사용한다.
    4. **시뮬레이션 프롬프트의 개방성:** 이 시스템의 핵심 목적은 오염되지 않은 원본 데이터를 구축하는 데 있다. 생성된 데이터를 다양한 사용 사례에 맞춰 어떻게 시스템 프롬프트에 동적으로 주입할지는 앱 개발자와 사용자의 자율에 맡긴다.

## 3. Database Schema (RDBMS + JSONB)

### 3-1. `users` (계정 및 식별)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `display_name` | String | 사용자 이름 |
| `identifier_tag` | String | 고유 식별 태그. 소셜 기능 활성화 시 사용. |
| `passcode_hash` | String | 셀프 호스팅 보안을 위한 PIN 또는 초대 토큰 |

### 3-2. `raw_store` (핵심 추출 데이터)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `user_id` | UUID | FK -> `users.id` |
| `method` | String | 추출 기법 |
| `domain_tag` | String | 해금 트리거용 메타데이터 (일/관계/자기) |
| `ts` | DateTime | 데이터 추출 타임스탬프 |
| `raw_data` | JSONB | **[가공 불가]** PII 치환 미들웨어를 거친 사용자 응답 원본 텍스트. |
| `prompt_version` | String | 사용된 시스템 프롬프트의 버저닝 해시. |
| `status` | String | 임시저장(`draft`) 또는 제출 완료(`completed`). |
| `is_used_for_validation` | Boolean | 프로브(정답)로 사용되었는지 여부. |

### 3-3. `domain_unlocks` (해금 상태 관리)
| Column | Type | Description |
|---|---|---|
| `user_id` | UUID | FK -> `users.id` |
| `domain` | String | 일 / 관계 / 자기 |
| `current_xp` | Integer | 획득한 현재 누적 경험치. 100 도달 시 검증 관문 해금. |
| `is_unlocked` | Boolean | 활용 모드가 열렸는지 여부. 기본값 `False`. |

### 3-4. `validation_sessions` (검증 세션 관리)
| Column | Type | Description |
|---|---|---|
| `session_id` | UUID | PK |
| `user_id` | UUID | FK -> `users.id` |
| `started_at` | DateTime | 검증 세션 시작 시각 |
| `contaminated` | Boolean | 활용 모드 피드백 등에 노출되어 오염되었는지 여부. |

### 3-5. `test_ledger` (검증 원장)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `user_id` | UUID | FK -> `users.id` |
| `test_type` | String | 검증 종류 |
| `prediction` | JSONB | LLM이 예측한 텍스트 또는 수치 |
| `ground_truth` | JSONB | 프로브 정답 |
| `discrepancy_score` | Float | 오차율 / 코사인 유사도 점수 |
| `prompt_version` | String | 평가 프롬프트의 버전. |
| `is_contaminated` | Boolean | 활용 모드 경험 이후에 치른 오염된 검증인지 여부. |

### 3-6. LangGraph Checkpoint Storage
LangGraph 내장 Checkpointer가 사용하는 표준 스키마를 따른다.

## 4. REST API Endpoints

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/auth/verify` | 인증 및 토큰 발급 |
| `GET` | `/api/user/raw_store` | 현재 사용자의 전체 원본 데이터 조회 |
| `POST` | `/api/extract/draft` | 진행 중인 검사 임시저장 |
| `POST` | `/api/extract/{method}` | 메서드 완료 및 데이터 적재 |
| `POST` | `/api/extract/ccrt` | CCRT 에피소드 일괄 전송 및 완료 처리 |
| `POST` | `/api/llm/facilitator` | 동적 메서드 수행 중 LLM 꼬리질문 생성 (SSE 비동기 스트리밍) |
| `POST` | `/api/validation/probe` | 검증 예측 생성 및 실제 응답 대조 |
| `POST` | `/api/social/feedback` | 익명 피드백 초대 링크 토큰 발급 |
| `POST` | `/api/utilize/chat` | 사용자 원본 데이터 기반 대화 턴 전송 |

## 5. Deployment Architecture

개인 데이터 보호를 위해 본 애플리케이션은 **Docker 기반 셀프 호스팅을** 공식 지원한다.

### 5-1. 기술 스택 예시
- **Frontend:** Next.js (또는 Vite + React)
- **Backend:** FastAPI (Python)
- **Database:** PostgreSQL 또는 단순 `raw_store.yaml`

### 5-2. Docker Compose 구조
로컬 머신 환경에서 쉽게 구동할 수 있도록 단일 `docker-compose.yml`을 제공한다. 데이터는 호스트 머신에 마운트되어 영구 보존된다.

## 6. Frontend Complex UI States

복합 UI 모듈(삼항 도출, 래더링) 렌더링 시 발생하는 지연을 방지하기 위해 다음 상태 관리 스펙을 따른다.

### 6-1. 삼항 도출 아키텍처
드래그 앤 드롭 및 다중 선택이 빈번하게 일어나는 뷰이므로, 서버 통신은 사용자가 모든 입력을 완료한 **마지막 단계에서 1회만** 수행한다.
- **흐름:** 3개 요소 선택 ➔ 2개를 유사 그룹으로, 1개를 이질 그룹으로 이동 ➔ 이유 텍스트 입력 ➔ 최종 페이로드 일괄 전송.

### 6-2. 래더링 아키텍처
꼬리질문이 깊어지는 재귀적 구조이므로 **카드가 위로 쌓이는 비동기 렌더링을** 적용한다.
- **흐름:** 사용자 답변 입력 ➔ 스택 최상단에 카드 Push ➔ 백엔드 호출 ➔ SSE 스트리밍으로 꼬리질문 실시간 업데이트 ➔ 종료 시 스택 전체 일괄 전송.
