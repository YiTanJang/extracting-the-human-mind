# App Blueprint: 심리 분석 서비스 아키텍처 명세서

> [!TIP]
> **To AI Coding Agents (Claude, Gemini, etc.):** 
> 이 문서는 [MVP 기획.md](<MVP 기획.md>)에 정의된 사용자 여정(User Journey)과 유저 플로우를 실제 소프트웨어(웹/앱)로 구현하기 위한 **엔지니어링 시스템 설계도**입니다. 기획적인 의도보다는 **Information Architecture (IA), 시스템 모듈 구성, 데이터 파이프라인, RDBMS 스키마** 구현에 집중하세요.

이 애플리케이션의 본질은 흔한 소셜 앱이 아닙니다. **"추출(Extraction) ➔ 검증(Validation) ➔ 활용(Utilization)"** 이라는 엄격한 단방향 파이프라인을 코드로 구현하는 것이 핵심입니다. 친구 추가 등의 소셜 기능은 단지 이 파이프라인의 '활용' 단계에서 데이터 소스를 주입하기 위한 선택적 확장(Add-on) 레이어일 뿐입니다.

---

## 1. Information Architecture (IA)

웹/앱의 라우팅 및 폴더 구조는 아래의 IA를 따릅니다. 사용자의 데이터 상태(해금 상태)에 따라 접근 권한이 통제되어야 합니다.

```text
/ (Root)
├── /onboarding             # 이름 기입, 고유 닉네임 발급, 실험 주의서 동의
├── /hub                    # [Module 1] 검사 허브. 11가지 추출 모듈 렌더링.
│   ├── /hub/ccrt
│   ├── /hub/laddering
│   └── ... (진행률 UI 포함)
├── /validation             # [Module 2] 검증 관문. 특정 도메인 임계치 도달 시 활성화.
│   └── /validation/a6      # Hold-out 예측 테스트 실행 뷰
├── /modes                  # [Module 3] 활용 모드 허브. 검증 통과 시 해금.
│   ├── /modes/chat         # 대화/상담 모드 UI (Chatbot)
│   ├── /modes/profiler     # 분석 모드 UI (Dashboard/Report)
│   └── /modes/sandbox      # 다중 에이전트 시뮬레이션 UI (Spectator View)
└── /social (Optional)      # 타인 데이터 주입 레이어. (고유 태그 연동 또는 로컬 파일 Import)
```

## 2. Core System Modules (시스템 플로우)

시스템은 3개의 독립적인 핵심 엔진으로 구성됩니다. 각 엔진은 앞선 엔진의 결과물만을 인풋으로 받습니다.

### Module 1: Extraction Engine (추출 엔진)
*   **Trigger:** 사용자가 `/hub`에서 특정 검사(예: CCRT) 시작.
*   **Logic:** LLM은 오직 정해진 프로토콜 질문만 던지도록 System Prompt로 통제받습니다. 사용자의 입력 텍스트는 **절대 LLM에 의해 요약, 환언, 라벨링되지 않고**, 날것(Verbatim) 그대로 JSON 파싱되어 DB(`raw_store`)에 Insert 됩니다.

### Module 2: Validation Gatekeeper (검증 엔진) - ★시스템의 코어★
*   **Trigger:** 특정 도메인의 데이터가 임계치(예: 3개)에 도달.
*   **Logic (Hold-out Prediction):**
    1.  서버 백엔드가 DB에서 해당 유저의 `raw_store` 데이터를 조회하되, **테스트용 타겟 데이터(Probe)를 마스킹(숨김) 처리**합니다.
    2.  숨기고 남은 데이터만 LLM에 주입하여 "이 사람은 타겟 질문에 어떻게 답했을까?"를 예측시킵니다.
    3.  프론트엔드 UI를 통해 사용자에게 실제 타겟 질문을 던져 **실제 정답**을 수집합니다.
    4.  백엔드에서 (LLM 예측값) vs (실제 정답)의 불일치율(Brier Score 등)을 연산하여 `test_ledger` 테이블에 기록합니다.
    5.  통과(또는 검증 완료) 시 해당 도메인의 `is_unlocked` 상태를 True로 업데이트합니다.

### Module 3: Utilization Engine (활용 엔진)
*   **Trigger:** 사용자가 `/modes`에서 특정 모드(분석, 시뮬레이션 등) 실행.
*   **Logic:**
    1.  `is_unlocked`가 True인 데이터만 조회합니다.
    2.  **데이터 소스 결합 (하이브리드 계층):** 만약 시뮬레이션에 타인(친구)을 포함했다면, 내 `raw_store` 배열과 친구의 `raw_store` 배열을 함께 가져옵니다. (이때 친구 데이터의 출처가 로컬 파일 Import인지, 서버 연동인지는 시스템 코어 입장에서 중요하지 않으며, 단지 JSON 배열 2개가 인풋으로 들어올 뿐입니다).
    3.  결합된 데이터를 기반으로 LLM 프롬프트를 빌드하여 활용 결과를 UI에 렌더링합니다.

## 3. Database Schema (RDBMS + JSONB)

기존 로컬 기반 `raw_store.yaml`과 `검증 원장`의 철학을 관계형 데이터베이스 모델로 매핑합니다. (PostgreSQL 권장).

### 3-1. `users` (계정 및 식별)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `display_name` | String | 사용자 이름 (앱 내 표기용) |
| `identifier_tag` | String | **(Optional)** 고유 식별 태그 (예: `User#1234`). 서버 연동 소셜 기능 활성화 시 사용. |

### 3-2. `raw_store` (핵심 추출 데이터)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `user_id` | UUID | FK -> `users.id` |
| `method` | String | 추출 기법 (예: `ccrt`, `triadic`) |
| `domain_tag` | String | 일 / 관계 / 자기 (해금 트리거용 메타데이터) |
| `raw_data` | JSONB | **[가공 불가]** 사용자 응답 원본 텍스트. (예: `{"wish": "인정", "ro": "무시", "rs": "화냄"}`) |
| `is_used_for_validation` | Boolean | 이 데이터가 Module 2(검증)의 프로브(정답)로 사용되었는지 여부. |

### 3-3. `domain_unlocks` (해금 상태 관리)
| Column | Type | Description |
|---|---|---|
| `user_id` | UUID | FK -> `users.id` |
| `domain` | String | 일 / 관계 / 자기 |
| `is_unlocked` | Boolean | 검증 관문을 통과하여 활용 모드가 열렸는지 여부. 기본값 `False`. |

### 3-4. `test_ledger` (검증 원장 - 연구 및 성능 측정 자산)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `user_id` | UUID | FK -> `users.id` |
| `test_type` | String | 검증 종류 (예: `A6_linguistic`, `B6_stress`) |
| `prediction` | JSONB | LLM이 예측한 텍스트 또는 수치 |
| `ground_truth` | JSONB | 실제 사용자의 응답 (프로브 정답) |
| `discrepancy_score` | Float | (예측 vs 실제) 오차율 / 코사인 유사도 / Brier 점수 |
| `is_contaminated` | Boolean | 활용 모드(LLM 피드백)를 경험한 **이후**에 치른 검증인지 여부. (오염 플래그) |
