# App Blueprint: 심리 분석 서비스 아키텍처 명세서

> **3줄 요약:**
> - 이 프로젝트가 실제 앱이나 웹 서비스로 만들어질 때 어떤 구조로 동작할지 보여주는 시스템 설계도입니다.
> - 데이터를 뽑아내고(추출), 맞는지 확인하고(검증), 이를 바탕으로 챗봇 상담이나 시뮬레이션을 하는(활용) 3단계 구조를 정의합니다.
> - 사용자의 개인정보를 보호하면서 데이터베이스에 어떻게 안전하게 저장할지(스키마) 명시합니다.

> [!TIP]
> **To AI Coding Agents (Claude, Gemini, etc.):** 
> 이 문서는 [MVP 기획.md](<MVP 기획.md>)에 정의된 사용자 여정(User Journey)과 유저 플로우를 실제 소프트웨어(웹/앱)로 구현하기 위한 **엔지니어링 시스템 설계도**이다. 기획적인 의도보다는 **Information Architecture (IA), 시스템 모듈 구성, 데이터 파이프라인, RDBMS 스키마** 구현에 집중한다.

이 애플리케이션의 본질은 흔한 소셜 앱이 아니다. **"추출(Extraction) ➔ 검증(Validation) ➔ 활용(Utilization)"** 이라는 엄격한 단방향 파이프라인을 코드로 구현하는 것이 핵심이다. 친구 추가 등의 소셜 기능은 단지 이 파이프라인의 '활용' 단계에서 데이터 소스를 주입하기 위한 선택적 확장(Add-on) 레이어일 뿐이다.

## 1. Information Architecture (IA)

웹/앱의 라우팅 및 폴더 구조는 아래의 IA를 따른다. 사용자의 데이터 상태(해금 상태)에 따라 접근 권한이 통제되어야 한다.

```text
/ (Root)
├── /onboarding             # 이름 기입, 고유 닉네임 발급, 실험 주의서 동의
├── /auth                   # [Auth] 초기 패스코드/초대 토큰 기반 인증 레이어
├── /hub                    # [Module 1] 검사 허브. 11가지 추출 모듈 렌더링.
│   ├── /hub/ccrt
│   ├── /hub/laddering
│   └── ... (진행률 UI 포함)
├── /validation             # [Module 2] 검증 관문. 특정 도메인 임계치 도달 시 활성화.
│    └── /validation/[test_id] # Hold-out 예측 테스트 실행 뷰 (예: a1, a6 등)
├── /modes                  # [Module 3] 활용 모드 허브. 검증 통과 시 해금.
│   ├── /modes/chat         # 대화/상담 모드 UI (Chatbot)
│   ├── /modes/profiler     # 분석 모드 UI (Dashboard/Report)
│   └── /modes/sandbox      # 다중 에이전트 시뮬레이션 UI (Spectator View)
└── /social (Optional)      # 타인 데이터 주입 레이어. 익명 피드백(조해리의 창) 초대 링크 발급 및 연동.
```

## 2. Core System Modules (시스템 플로우)

시스템은 3개의 독립적인 핵심 엔진으로 구성된다. 각 엔진은 앞선 엔진의 결과물만을 인풋으로 받는다.

### Module 1: Extraction Engine (추출 엔진)
*   **Trigger:** 사용자가 `/hub`에서 특정 검사(예: CCRT, 래더링) 시작.
*   **Logic:** 검사의 성격에 따라 두 가지 파이프라인으로 나뉜다.
    1.  **정적 검사 (CCRT 등):** LLM이 개입하지 **않는다**. 앱은 정적인 위키 질문을 화면에 띄워주고, 사용자는 빈 텍스트 박스에 혼자 생각한 답을 적어 DB로 직행시킨다.
    2.  **동적 검사 (래더링 등):** 지속적인 꼬리질문("왜 중요하죠?")이 필요하므로 LLM(챗봇)이 보조(Facilitator)로 투입된다. 단, System Prompt를 통해 **"절대 사용자의 의도를 지레짐작해 요약/라벨링하지 말고 오직 꼬리질문만 할 것"**을 강제한다.
*   **데이터 적재 원칙 (Raw Data Preservation):** 정적/동적 방식 모두 최종적으로 사용자가 입력한 텍스트는 **절대 요약되거나 환언되지 않고**, 날것(Verbatim) 그대로 구조화되어 DB(`raw_store`)에 Insert 된다.
*   **전역 가명화 (PII Scrubbing Middleware):** 단, 프라이버시 보호를 위해 DB 저장 및 외부 LLM API 전송 직전에 **정규표현식 및 NER(Named Entity Recognition) 기반의 가명화 미들웨어**를 반드시 거친다. 
    *   **핵심 원칙:** LLM에게 "문장을 가명화해줘"라고 지시하면 문장의 어투나 구조를 마음대로 요약·변형할 위험이 크므로, **문장의 다른 어떤 글자도 건드리지 않고 오직 실명(사람 이름, 직장명 등)만 `[Person_A]`, `[Company_B]` 등으로 치환**하는 결정론적(Deterministic) 필터링을 강제한다.
*   **LLM 하이퍼파라미터 (Temperature 표준화):** 
    *   추출 봇(Module 1) 및 검증 채점기(Module 2): 환각을 막고 일관된 분석을 위해 **Temperature = 0.0 ~ 0.2**로 강제한다.
*   **JSON Schema (Pydantic 예시):** 프론트엔드 Form(또는 챗봇 로그 파서)에서 DB로 넘어가는 데이터 규격이다.
    ```python
    class CCRTEntry(BaseModel):
        id: str
        method: Literal["ccrt"]
        object_tag: str
        raw_wish: str
        raw_response_other: str
        raw_response_self: str
        raw_recurrence: str
    ```

### Module 2: Validation Gatekeeper (검증 엔진) - ★시스템의 코어★
*   **Trigger:** 특정 도메인의 데이터가 임계치(예: 3개)에 도달.
*   **Logic (Hold-out Prediction):**
    1.  서버 백엔드가 DB에서 해당 유저의 `raw_store` 데이터를 조회하되, **테스트용 타겟 데이터(Probe)를 마스킹(숨김) 처리**한다.
    2.  숨기고 남은 데이터만 LLM에 주입하여 "이 사람은 타겟 질문에 어떻게 답했을까?"를 예측시킨다.
    3.  프론트엔드 UI를 통해 사용자에게 실제 타겟 질문을 던져 **실제 정답**을 수집한다.
    4.  **채점 알고리즘 (Cross-Encoder STS):** 물리적으로 분리된 평가용 STS(Semantic Textual Similarity) 모델(예: Cross-Encoder)을 호출하여 두 텍스트 간 논리적 일치도를 0~1 스칼라 값으로 직접 출력받고 `test_ledger`에 기록한다.
    5.  최소 배터리(기본 hold-out 예측 1회 + 라인업 1라운드)를 오염 없이 적재 완료 시 해당 도메인의 `is_unlocked` 상태를 True로 업데이트한다. **정확도 점수가 낮아도 통과** — 낮은 점수도 방법론 증거 데이터다. (→ `검증 모드.md` §2-2)

### Module 3: Utilization Engine (활용 엔진)
*   **Trigger:** 사용자가 `/modes`에서 특정 모드(분석, 시뮬레이션 등) 실행.
*   **Logic:**
    1.  `is_unlocked`가 True인 데이터만 조회한다.
    2.  **데이터 소스 결합 (하이브리드 계층):** 만약 시뮬레이션에 타인(친구)을 포함했다면, 내 `raw_store` 배열과 친구의 `raw_store` 배열을 함께 가져온다. (이때 친구 데이터의 출처가 로컬 파일 Import인지, 서버 연동인지는 시스템 코어 입장에서 중요하지 않으며, 단지 JSON 배열 2개가 인풋으로 들어올 뿐이다).
    3.  **멀티 에이전트 아키텍처:** 다중 에이전트 시뮬레이션의 경우, 1개의 통합된 프롬프트에 나+친구 데이터를 모두 때려 넣고 LLM이 혼자 1인 2역 소설을 쓰게 해서는 안 된다 (토큰 오버플로우 및 자아 혼란 발생). 반드시 **독립된 LLM 인스턴스(Agent A, Agent B)를 각각 메모리에 띄우고**, 그 위에 턴(Turn)을 강제로 통제하고 위기 상황(Flip Condition)을 강제 주입하는 중앙 통제기(**Environment State Machine / 판사 노드**) 계층을 씌운 구조로 구현해야 한다. 
        * 이 시뮬레이션 에이전트들의 하이퍼파라미터는 자연스럽고 창의적인 롤플레잉 대화를 유도하기 위해 **Temperature = 0.7**을 기본값으로 사용한다.
    4.  **시뮬레이션 프롬프트의 개방성(Open-endedness):** 이 저장소에 구체적인 "시뮬레이션 프롬프트 템플릿"이 하드코딩되어 있지 않은 것은 블랙박스가 아니라 의도된 설계이다. 이 프로젝트의 핵심 목적은 '오염되지 않은 완벽한 `raw_store.yaml`을 구축하는 것'까지이다. 완성된 YAML 데이터를 갈등 중재, 진로 상담, 롤플레잉 등 무한한 Use-case에 맞춰 어떤 시스템 프롬프트에 동적으로 주입(Inject)할지는 온전히 앱 개발자와 사용자의 자유도에 맡긴다.

## 3. Database Schema (RDBMS + JSONB)

기존 로컬 기반 `raw_store.yaml`과 `검증 원장`의 철학을 관계형 데이터베이스 모델로 매핑한다. (PostgreSQL 권장).

### 3-1. `users` (계정 및 식별)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `display_name` | String | 사용자 이름 (앱 내 표기용) |
| `identifier_tag` | String | **(Optional)** 고유 식별 태그 (예: `User#1234`). 서버 연동 소셜 기능 활성화 시 사용. |
| `passcode_hash` | String | 셀프 호스팅 보안을 위한 사용자별 4자리 PIN 또는 초기 발급 초대 토큰 (D-4-1) |

### 3-2. `raw_store` (핵심 추출 데이터)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `user_id` | UUID | FK -> `users.id` |
| `method` | String | 추출 기법 (예: `ccrt`, `triadic`) |
| `domain_tag` | String | 일 / 관계 / 자기 (해금 트리거용 메타데이터) |
| `ts` | DateTime | 데이터가 추출된 타임스탬프 (시계열 분석 및 컨텍스트 로딩 용도) |
| `raw_data` | JSONB | **[가공 불가]** PII 치환 미들웨어를 거친 사용자 응답 원본 텍스트. (예: `{"wish": "인정", "ro": "무시", "rs": "화냄"}`) |
| `prompt_version` | String | (Optional) 이 데이터를 추출할 때 사용된 시스템 프롬프트의 해시/버전 명시 (E-1 버저닝). |
| `status` | String | `draft` (임시저장) 또는 `completed` (제출 완료). (E-2 세션 복구용) |
| `is_used_for_validation` | Boolean | 이 데이터가 Module 2(검증)의 프로브(정답)로 사용되었는지 여부. |

### 3-3. `domain_unlocks` (해금 상태 관리)
| Column | Type | Description |
|---|---|---|
| `user_id` | UUID | FK -> `users.id` |
| `domain` | String | 일 / 관계 / 자기 |
| `current_xp` | Integer | (E-2) 해당 도메인에서 획득한 현재 누적 경험치. 100 도달 시 검증 관문 해금. |
| `is_unlocked` | Boolean | 검증 관문을 통과하여 활용 모드가 열렸는지 여부. 기본값 `False`. |

### 3-4. `validation_sessions` (검증 세션 관리)
| Column | Type | Description |
|---|---|---|
| `session_id` | UUID | PK |
| `user_id` | UUID | FK -> `users.id` |
| `started_at` | DateTime | 검증 세션 시작 시각 |
| `contaminated` | Boolean | 세션 진행 중 활용 모드 피드백 등에 노출되어 오염되었는지 여부. (브라우저 이탈 시 복구용) |

### 3-5. `test_ledger` (검증 원장 - 연구 및 성능 측정 자산)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | PK |
| `user_id` | UUID | FK -> `users.id` |
| `test_type` | String | 검증 종류 (예: `A6_linguistic`, `B6_stress`) |
| `prediction` | JSONB | LLM이 예측한 텍스트 또는 수치 |
| `ground_truth` | JSONB | 실제 사용자의 응답 (프로브 정답) |
| `discrepancy_score` | Float | (예측 vs 실제) 오차율 / 코사인 유사도 / Brier 점수 |
| `prompt_version` | String | 이 검증을 채점(LLM-as-a-judge)할 때 사용된 평가 프롬프트의 해시/버전. |
| `is_contaminated` | Boolean | 활용 모드(LLM 피드백)를 경험한 **이후**에 치른 검증인지 여부. (오염 플래그) |

### 3-6. LangGraph Checkpoint Storage (시뮬레이션 상태 영구 저장용)
LangGraph의 `AsyncPostgresSaver` 등 내장 Checkpointer가 사용하는 표준 스키마를 준수한다.
| Table | Column | Type | Description |
|---|---|---|---|
| `checkpoints` | `thread_id` | String | 시뮬레이션 세션의 고유 스레드 ID |
| `checkpoints` | `checkpoint_ns` | String | 체크포인트 네임스페이스 |
| `checkpoints` | `checkpoint_id` | String | 해당 턴의 체크포인트 고유 식별자 |
| `checkpoints` | `checkpoint` | JSONB | 에이전트 그래프의 전체 State (메시지 이력, 커플링 수치 등) 영구 저장 |
| `checkpoint_writes` | `thread_id` | String | |
| `checkpoint_writes` | `checkpoint_id` | String | |
| `checkpoint_writes` | `task_id` | String | 그래프 내 비동기 태스크 식별자 |
| `checkpoint_writes` | `channel` | String | LangGraph Channel 이름 |
| `checkpoint_writes` | `type` | String | 작성 유형 |
| `checkpoint_writes` | `value` | JSONB | 채널에 기록되는 개별 메시지/상태 값 |

## 4. REST API Endpoints (Contract)

프론트엔드와 백엔드 분리 구조(예: Next.js + FastAPI)를 상정한 기본 API 명세이다.

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/auth/verify` | 사용자 식별 태그 및 패스코드(PIN) 확인 후 세션/JWT 토큰 발급 |
| `GET` | `/api/user/raw_store` | 현재 로그인된 사용자의 전체 `raw_store` 데이터 조회 |
| `POST` | `/api/extract/draft` | 진행 중인 검사의 JSON Payload를 임시저장(DB Upsert, status=draft) |
| `POST` | `/api/extract/{method}` | 특정 메서드 수행 결과 객체를 DB에 Append (status=completed 변환 및 XP 보상 처리) |
| `POST` | `/api/extract/ccrt` | CCRT의 경우, 로컬 스토리지에 모아둔 에피소드 배열(`List[CCRTEntry]`, 최소 3장)을 한 번에 전송하여 완료 처리 |
| `POST` | `/api/llm/facilitator` | 래더링 등 동적 메서드 수행 중 LLM 꼬리질문 생성 (SSE 비동기 스트리밍 반환) |
| `POST` | `/api/validation/probe` | 검증 모드 진입 시, LLM의 사전 예측(봉인) 및 사용자 실제 응답 대조 |
| `POST` | `/api/validation/calendar` | 기기 캘린더 연동을 위한 OAuth 권한 요청 및 특정 미래 사건 타겟팅 스케줄링 |
| `POST` | `/api/social/feedback` | 실제 지인에게 전송할 수 있는 익명 피드백 초대 링크 토큰 발급 및 수집 |
| `POST` | `/api/utilize/chat` | 챗봇 모드 등에서 사용자 원본 데이터를 기반으로 한 대화 턴 전송 |

## 5. Deployment Architecture (Self-Hosting)

개인 데이터 보호를 위해 본 애플리케이션은 **Docker 기반 셀프 호스팅(Self-Hosting)** 을 공식 지원한다.

### 5-1. 기술 스택 (예시 권장)
*   **Frontend:** Next.js (또는 Vite + React)
*   **Backend:** FastAPI (Python, LLM 오케스트레이션 특화)
*   **Database:** 로컬 기반 SQLite, PostgreSQL, 또는 단순 `raw_store.yaml` 파일 시스템 입출력

### 5-2. Docker Compose 구조
로컬 머신이나 개인 NAS 환경에서 쉽게 구동할 수 있도록 단일 `docker-compose.yml`을 제공한다. 데이터는 호스트 머신의 `./data` 폴더에 마운트되어 컨테이너가 파괴되어도 영구 보존된다.

```yaml
# 예시 구조
services:
  frontend:
    build: ./frontend
    ports: ["3000:3000"]
  backend:
    build: ./backend
    ports: ["8000:8000"]
    volumes:
      - ./data:/app/data   # 로컬 원장 데이터 영구 보존
    environment:
      - OPENAI_API_KEY=${OPENAI_API_KEY}
      - ADMIN_PASSCODE=${ADMIN_PASSCODE} # 셀프 호스트용 단일 패스워드
```

## 6. Frontend Complex UI States (E-5 명세)

프론트엔드(React/Next.js 등) 개발 시 잦은 렌더링 지연(Lag)을 방지하고 부드러운 상호작용을 보장하기 위해, 복합 UI 모듈(삼항 도출, 래더링)은 아래의 로컬 상태(Local State) 관리 스펙을 따른다.

### 6-1. 삼항 도출 (Triadic Elicitation) 아키텍처
드래그 앤 드롭 및 다중 선택이 빈번하게 일어나는 뷰이므로, 서버와의 통신은 사용자가 모든 입력을 완료한 **마지막 단계에서 단 1회**만 수행한다.
*   **상태 정의 (Local State):**
    *   `selectedItems: string[]` (최대 3개 제한)
    *   `similarPair: string[]` (2개 할당용)
    *   `diffItem: string | null` (1개 할당용)
    *   `constructs: { similarReason: string, diffReason: string }`
*   **흐름 (Flow):**
    1. 요소 풀(Pool)에서 3개 요소 클릭 선택 ➔ 상태 업데이트.
    2. 드래그 앤 드롭 인터페이스 ➔ 2개를 '유사 그룹', 1개를 '이질 그룹' 영역으로 이동시켜 상태 할당.
    3. 텍스트 입력창 활성화 ➔ 왜 유사하고, 왜 다른지 입력.
    4. **Submit:** `POST /api/extract/triadic`으로 최종 조합된 JSON 페이로드를 일괄 전송.

### 6-2. 래더링 (Laddering) 아키텍처
꼬리질문이 무한히 깊어질 수 있는 재귀적(Recursive) 구조이므로, 전통적인 채팅창 UI 대신 **카드가 위로 쌓이는(Stacking)** 시각적 은유를 사용하며 비동기 렌더링을 적용한다.
*   **상태 정의 (Local State):**
    *   `ladderStack: Array<{ id: string, question: string, answer: string }>`
    *   `isStreaming: boolean`
*   **흐름 (Flow):**
    1. 사용자가 답변을 입력하면, 새 노드가 `ladderStack` 최상단에 Push되고 화면 상단에 추가된다. 기존 카드들은 아래로 밀려 스크롤된다. (가장 최근 답변 = 화면 최상단, `래더링.md` §3 UX 스펙 기준)
    2. 백엔드의 `POST /api/llm/facilitator`를 호출한다.
    3. **SSE (Server-Sent Events) 스트리밍:** 백엔드에서 생성되는 꼬리질문을 SSE로 받아, 스택 최상단 카드(= 방금 추가된 카드)의 `question` 속성을 실시간으로 업데이트(Typing Effect)한다.
    4. 사용자가 특정 노드에서 더 이상 답할 수 없어 "종료"를 누르면, 스택 전체 데이터를 묶어 `POST /api/extract/laddering`에 제출하고 완료 처리한다.
