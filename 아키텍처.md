# App Blueprint: 심리 분석 서비스 아키텍처 청사진

> [!TIP]
> **To AI Coding Agents (Claude, Gemini, etc.):** 
> 이 문서는 심리 추출 방법론 위키를 기반으로 **실제 소프트웨어**를 구현하기 위한 거시적인 설계도입니다. 개발 스케일에 따라 아래의 두 가지 아키텍처 모델 중 하나를 선택하여 코드를 생성하세요. 복잡하고 무거운 기술 스택보다는 보편적인 스택(Next.js, Python, Supabase 등)을 가정합니다.

이 프로젝트는 본질적으로 'LLM에게 줄 데이터를 수집하는 프로토콜'입니다. 이를 어떻게 포장하느냐에 따라 가벼운 로컬 프로그램이 될 수도 있고, 거대한 소셜 플랫폼이 될 수도 있습니다.

---

## Model A. 로컬 기반 개인용 프로그램 (Local Standalone App)
서버 호스팅이나 복잡한 DB 구축이 귀찮을 때 가장 빠르게 만들 수 있는 형태입니다. 모든 데이터는 사용자의 기기 안(Local)에만 머무릅니다.

*   **추천 스택:** Electron, Tauri, 혹은 가벼운 로컬 Next.js/Python 웹 뷰 + SQLite (또는 Local JSON/YAML)
*   **핵심 작동 방식:**
    1.  **본인 데이터 수집:** 사용자는 앱 내 챗봇과 대화하며 자신의 심리 데이터(CCRT, 삼항 도출 등)를 추출하여 로컬 파일에 저장합니다.
    2.  **친구 데이터 수동 추가 (Manual Import):** 서버 기반의 친구 추가 기능이 없습니다. 대신 **"다른 사람 데이터 추가하기"** 버튼이 존재합니다. 지인이 다른 곳에서 추출한 자신의 `raw_store.yaml` 파일이나 텍스트 뭉치를 메신저로 보내주면, 사용자가 이를 앱에 수동으로 복사·붙여넣기(Import) 하여 타인의 프로필을 생성합니다.
    3.  **로컬 멀티 시뮬레이션:** 앱은 내 데이터와 방금 추가한 타인의 데이터를 LLM API에 동시에 주입하여 궁합을 보거나 가상 갈등을 시뮬레이션합니다.

## Model B. 고유 닉네임 태그 기반 소셜 플랫폼 (Hosted Social Service)
웹 상에 호스팅되어 다수의 사용자가 가입하고 상호작용하는 본격적인 SaaS 형태의 플랫폼 아키텍처입니다.

*   **추천 스택:** Next.js (프론트) + Node.js / FastAPI (백엔드) + PostgreSQL / Supabase (DB)
*   **핵심 작동 방식 (큰 골조):**
    1.  **식별 태그 발급:** 실명이나 이메일을 통한 검색을 지양하고, 가입 시 고유한 익명 식별자(예: `User#1234`)를 발급합니다.
    2.  **상호 동의 기반 연결 (Friend System):** 고유 태그를 교환하여 친구 추가 요청을 보냅니다. 양측이 수락하면 서버 상에서 두 유저의 심리 데이터가 '연동 가능 상태'가 됩니다.
    3.  **제로-널리지(Zero-Knowledge) 시뮬레이션:**
        *   친구가 되었다고 해서 서로의 `raw_data`(가장 내밀한 과거 기억이나 두려운 자기 등)를 직접 열람할 수 있는 것은 **절대 아닙니다.**
        *   대신, 플랫폼 내에서 "나와 User#1234의 연인 궁합 보기" 또는 "좀비 사태에서 둘의 생존 시뮬레이션" 버튼을 누를 수 있습니다.
        *   백엔드 서버가 DB에서 두 사람의 데이터를 은밀하게 모아 LLM 컨텍스트에 주입하고, **LLM이 연산해 낸 '결과 텍스트'만을 사용자들에게 보여줍니다.** (개인정보는 서버-LLM 사이에서만 처리됨).

## 공통 구현 핵심 (Core Features)

어떤 아키텍처를 선택하든 앱 내부에는 다음의 기능들이 필수로 구현되어야 합니다.

### 1. 추출 모드 (Extraction Hub)
*   **UI:** 챗봇(Chat) 인터페이스.
*   **로직:** 사용자가 폼을 채우는 것이 아니라, LLM이 시스템 프롬프트(이 위키의 추출 문서들)를 바탕으로 사용자에게 질문을 던집니다. 사용자의 응답은 LLM이 절대로 임의 요약하거나 해석하지 못하게 막고, 날것 그대로(Verbatim) DB나 파일에 저장해야 합니다.

### 2. 프로파일링 모드 (Analytics View)
*   **UI:** 대시보드 리포트.
*   **로직:** 저장된 날것의 데이터들을 모아 LLM에게 분석을 요청합니다. LLM은 사용자의 가치관, 모순, 핵심 갈등 패턴을 텍스트로 요약해 보여줍니다. 

### 3. 멀티 에이전트 샌드박스 (Sandbox)
*   **UI:** 관전형(Spectator) 대화창 또는 상호작용형 시뮬레이션 룸.
*   **로직:** 내 데이터로 만들어진 '디지털 트윈'과 타인의 '디지털 트윈'을 가상의 딜레마 상황에 던져놓고 관전하거나, 내가 직접 타인의 트윈과 대화하며 갈등 해결을 미리 연습해 볼 수 있는 공간입니다.
