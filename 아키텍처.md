# App Blueprint: 심리 분석 서비스 아키텍처 청사진

> [!TIP]
> **To AI Coding Agents (Claude, Gemini, etc.):** 
> 이 문서는 심리 추출 방법론 위키를 실제 동작하는 Full-stack 웹 어플리케이션으로 구현하기 위한 **소프트웨어 설계 명세서(Blueprint)**입니다. 이 문서를 읽고 지시받은 언어/프레임워크(예: Next.js, FastAPI, Supabase 등)를 사용하여 코드를 생성하세요.

## 1. System Architecture (권장 스택)

*   **Frontend:** Next.js (React), Tailwind CSS (채팅형 UI 및 대시보드 렌더링)
*   **Backend:** Node.js (Express/NestJS) 또는 Python (FastAPI). LLM 통신과 DB 저장을 담당.
*   **Database:** PostgreSQL (Supabase 권장). 로컬 `raw_store.yaml`을 대체할 관계형 DB.
*   **AI Layer:** OpenAI API (GPT-4o) 또는 Gemini API. 사용자 입력 유도(추출) 및 데이터 분석(활용).

---

## 2. Database Schema (데이터 모델)

기존 `raw_store.yaml`의 철학을 그대로 RDBMS 스키마로 변환합니다. 모든 데이터는 사후 가공(해석) 없이 날것(Verbatim)으로 저장되어야 합니다.

### Table: `users`
| Column | Type | Description |
|---|---|---|
| `id` | UUID | Primary Key |
| `created_at` | Timestamp | 가입일 |

### Table: `sessions` (검사 세션 관리)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | Primary Key |
| `user_id` | UUID | Foreign Key -> `users.id` |
| `method` | String | 추출 기법 이름 (예: `ccrt`, `laddering`, `triadic`) |
| `status` | String | `IN_PROGRESS`, `COMPLETED` |

### Table: `raw_entries` (핵심 데이터 저장소)
| Column | Type | Description |
|---|---|---|
| `id` | UUID | Primary Key |
| `session_id` | UUID | Foreign Key -> `sessions.id` |
| `user_id` | UUID | Foreign Key -> `users.id` |
| `method` | String | 추출 기법 (예: `ccrt`) |
| `object_tag` | String | 관계 유형 (예: "동료", "어머니" - **실명 절대 금지**) |
| `raw_data` | JSONB | 사용자 응답 원본. (예: `{"raw_wish": "인정받고 싶음", "raw_ro": "무시함", "raw_rs": "화냄"}`) |
| `created_at` | Timestamp | 데이터 생성일 |

> [!WARNING]
> `raw_entries.raw_data` 컬럼에 데이터를 넣을 때, **절대 백엔드나 LLM이 요약, 환언, 라벨링을 해서는 안 됩니다.** 프론트엔드에서 사용자가 입력한 텍스트 그대로(`verbatim`) Insert 해야 합니다.

## 3. Core Workflows (핵심 API 플로우)

### Flow 1. 추출 모드 (Extraction Flow - 챗봇 형태)
사용자가 직접 폼(Form)을 채울 수도 있지만, LLM 챗봇이 자연스럽게 질문을 던져서 3항 도식을 뜯어내는 구조가 UX에 유리합니다.

1.  **[Client]** 사용자가 'CCRT 검사 시작' 버튼 클릭.
2.  **[Server]** DB에 새로운 `sessions` 레코드 생성 후, LLM API에 `핵심 갈등 도식 (CCRT).md`의 프롬프트를 System Prompt로 주입.
3.  **[AI Layer]** LLM이 "최근 부딪힘이 반복되는 사람의 관계 유형(예: 동료)을 알려주세요"라고 첫 질문 발송.
4.  **[Client/Server]** 사용자의 답변 핑퐁. LLM은 반드시 1. 소망 -> 2. 상대 반응 -> 3. 내 반응 순서대로 질문해야 함 (Function Calling 활용 권장).
5.  **[Server]** 3가지 요소를 모두 수집하면, 백엔드가 `raw_entries` 테이블에 JSONB 형태로 Insert 하고 세션 종료.

### Flow 2. 분석 모드 (Analysis & Profiling Flow)
1.  **[Client]** 사용자가 대시보드에서 "내 인간관계 패턴 분석하기" 클릭.
2.  **[Server]** `raw_entries` 테이블에서 해당 `user_id`의 데이터 10건을 전부 `SELECT`.
3.  **[Server]** 서버 내부 프롬프트(`analysis_prompt.md` 참고) 템플릿에 조회한 JSON 배열을 통째로 주입하여 LLM API 호출.
    *   *예: "이 유저의 raw 데이터를 보고, 반복되는 갈등 패턴을 찾아라. 단, 데이터에 없는 내용을 지어내지 마라."*
4.  **[Client]** 분석 결과를 UI에 렌더링. (이 분석 결과는 DB에 캐싱해두어도 되지만, 원본 raw_data를 덮어쓰거나 오염시켜서는 안 됨).

## 4. UI / UX Wireframe Specs

*   **View 1. Dashboard (대시보드):** 
    *   현재 나의 누적 `raw_entries` 갯수 시각화.
    *   "새로운 기억 기록하기" FAB (Floating Action Button).
*   **View 2. Extraction Hub (검사 허브):**
    *   채팅 인터페이스(Chat UI). 하단에 메시지 입력창 존재. 
    *   상단에 현재 진행 중인 검사 프로토콜(예: "진행률: [소망] 완료 -> [타인 반응] 대기 중") 표시.
*   **View 3. Insights (분석 결과 뷰):**
    *   LLM이 생성한 패턴 리포트 표시.
    *   리포트 내 특정 문장을 클릭하면, 그 결론의 근거가 된 과거의 `raw_data` (예: 3주 전의 직장 상사 관련 응답)가 모달(Modal)로 뜨면서 출처 추적성(Traceability) 제공.

---

## 5. Security & Privacy (가장 중요한 구현 원칙)

1.  **익명화 강제:** 프론트엔드 입력창에서 사용자에게 대상의 실명(이름)을 묻는 필드를 만들지 마세요. 무조건 `Dropdown`이나 `Radio button`으로 관계 유형(가족/친구/동료/연인 등)만 선택하게 해야 합니다.
2.  **프롬프트 누수 방지:** 백엔드에서 LLM으로 raw data를 쏠 때, 개인 식별 정보(PII)가 섞여 들어가지 않도록 정규식(Regex) 필터링 파이프라인을 추가하는 것을 권장합니다.
