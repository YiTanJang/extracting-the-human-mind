# 기여 가이드 (Contributing)

이 저장소는 **방법론·기획 문서 위키**이다. 코드가 아니라 *아이디어와 설계*에 기여한다.

## 무엇을 환영하나

- **반박·의심** — "이 추출 방법은 LLM이 해석 못 할 것 같다", "이 검증 설계는 looping에 오염된다" 같은 비판. 이 프로젝트는 [검증 모드](<검증 모드.md>)의 정신대로 *틀렸을 가능성*을 환영한다.
- **새 추출 기법** — 심리학 이론에 근거한 raw 추출 방법 제안.
- **활용·검증 설계** — 모드 개선, 검증 프로토콜·베이스라인 구성.
- **구현 사례** — 이 방법론으로 무언가를 만들었다면 링크 공유.

## 어떻게

- **가벼운 의견·질문** → [Issue](../../issues).
- **문서 수정·추가** → Fork → 편집 → Pull Request.

## 문서 구조 가이드

새로운 추출 모듈이나 문서를 추가할 때는 아래의 템플릿을 따른다.

## 1. 헤더 (Header)

모든 문서 상단에는 요약 블록을 포함한다.

```markdown
# Methodology: [모듈 이름]

> **예상 소요 시간:** O~O분 (해당될 경우)
> **세 줄 요약:**
> - (비전문가를 위한 쉬운 설명 1)
> - (비전문가를 위한 쉬운 설명 2)
>
>
> **설계 핵심:**
> - **목적/배경:** (이론적 연원이나 핵심 목적)
> - **특징:** (기타 기술적 특징)
```
## 0. 이론적 전제 및 설계 원칙
## 단계 1..N         # 각 단계: 근거(방법론) → UI 프롬프트 → 동작/산출
## 단계 N. 산출물 및 데이터 저장 (Raw Store Append)   # YAML 예시
## 단계 N. 알려진 한계 (Limitations)

### 활용 모드 문서

```
# 모드명 (English)
> **한 줄 정체성:** ...

## 1. 방법론 (Methodology)
## 2. UI · 상호작용 설계 (Interaction)
## 3. 사용 예시 (Use Cases)
## 4. 규칙 (Behavioral Rules)
## 5. 알려진 한계 (Limitations)
```

### 핵심 원칙 (어기지 말 것)

- **raw는 verbatim.** 정규화·요약·라벨링 금지. 측정 메타데이터만 붙인다.
- **무오염.** 설계자 메뉴·유도 질문 금지. 빈칸으로 받는다.
- **추출/활용 분리.** 해석·점수·분류는 전부 PART 2(활용)에 둔다. raw_store엔 해석 필드를 두지 않는다.
- **링크 형식.** 문서 간 링크는 `[문서명](<문서명.md>)` (앵글 브래킷) 형식. GitHub·Obsidian 양쪽에서 동작한다.
