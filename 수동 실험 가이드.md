# 수동 실험 가이드

## 시작하기 전에

> **안내**
> **이 프로젝트는 앱 서비스 형태의 소프트웨어가 아닌, 방법론 위키이다.**
> 
> *앱 구조가 궁금하다면 **[아키텍처](<아키텍처.md>)** 문서를 참조한다.*
> 
> *(※ 단, 이론적 증명을 목적으로 종이와 펜, 파이썬 환경, LLM을 이용해 수동으로 심리 추출의 전 과정을 수행해 볼 수 있다. 앱 기반의 자동화가 없기 때문에 수동으로 조작해야 한다.)*

이 가이드는 수동 환경에서 챗봇(LLM)과 파이썬 코드를 교차로 통제하여, 자신의 심리 데이터를 오염 없이 추출하는 **수동 오케스트레이션** 방법을 설명한다.

---

## 준비물

수동 실험을 위해 다음 세 가지 환경을 준비한다.

1. **로컬 텍스트 에디터 (메모장 등):** 
   - `raw_store.yaml` 파일을 생성하여 본인의 원본 텍스트를 보관한다.
2. **LLM 챗봇 (ChatGPT, Claude 등):** 
   - 동적 검사(래더링) 시 꼬리질문을 던져주는 '퍼실리테이터' 역할을 담당한다.
3. **파이썬 실행 환경 (Google Colab, Jupyter Notebook 등):** 
   - 복잡한 수치 연산을 직접 실행하기 위한 환경이다. 웹 브라우저에서 바로 실행 가능한 **Google Colab**을 권장한다.

---

## 핵심 원칙: 추출 단계의 LLM 개입 통제

> **경고**
> **데이터를 추출할 때 LLM에게 답변의 분석이나 요약을 맡기지 않는다.**
> "내 대답을 분석해서 점수를 매겨줘"라고 요청하면, LLM은 은연중에 편향을 섞거나 환각을 통해 잘못된 결과를 낸다. 이 실험의 핵심은 **개입 없는 순수한 원본 데이터의 확보이다**.

검사의 성격에 따라 다음 두 가지 방식으로 진행한다.

**1. 정적 검사 (예: CCRT, 두려운 자기)**
- 질문이 고정되어 있으므로 **LLM을 사용하지 않는다.** 생각한 답을 텍스트 에디터의 `raw_store.yaml` 파일에 바로 적는다.

**2. 동적 검사 (예: 래더링)**
- 원인과 이유를 깊게 파고들기 위해 **LLM의 꼬리질문 보조가 필수적이다.**
- 단, LLM은 엄격하게 훈련된 **단순한 질문 생성기** 역할만 수행해야 한다. 프롬프트로 답변 요약을 차단한 후, 대화가 끝나면 **사용자가 입력한 원본 텍스트만 발췌하여** 저장한다.

수동 환경에서는 백엔드 서버의 자동화가 없으므로 **사용자가 직접 각 단계를 통제해야 한다.**

- **수식 연산 및 결과 판정:** 각 방법론 문서 하단에 제공된 파이썬 코드를 실행하여 결과를 얻는다.
- **꼬리질문 생성:** LLM 챗봇을 활용한다.

---

## 워크플로우 예시: 삼항 도출에서 래더링으로

가장 복잡한 '삼항 도출 → 래더링' 파이프라인의 수동 진행 예시이다.

### Phase 1. 삼항 도출 (수동 기록)
1. **[삼항 도출](<삼항 도출.md>)** 문서를 읽고 9개의 구성 요소를 선정한다.
2. 원소를 3개씩 묶어 "2개는 비슷하고 1개는 다른 이유"를 스스로 작성한다. 이 과정에서 LLM은 개입하지 않는다.
3. 텍스트 에디터의 `raw_store.yaml` 파일에 직접 설정한 축과 평점 행렬을 기입한다.

### Phase 2. 수학적 타당성 계산 (Python)
1. **Google Colab** 환경을 준비한다.
2. **[삼항 도출](<삼항 도출.md>)** 문서의 파이썬 코드 블록을 Colab으로 복사하여 실행한다.
3. `raw_store.yaml`에 기록한 평점 행렬 데이터를 입력값으로 넣어 파이썬 코드를 실행한다.
4. **결과 확인:** 파이썬 실행 결과에서 가장 편차가 높은 1위 타겟 축을 확인한다.

### Phase 3. 챗봇에 타겟 주입 (LLM)
1. 1위 타겟을 가지고 **ChatGPT/Claude** 창으로 이동한다.
2. **[래더링](<래더링.md>)** 문서의 '시스템 프롬프트'를 복사하여 챗봇에게 입력한다.
3. 타겟 가치를 제시하며 래더링 꼬리질문을 시작할 것을 요청한다.
4. 챗봇이 던지는 "왜 그것이 중요한가?"라는 질문에 반복해서 답변한다.

### Phase 4. 최종 적재
대화가 깊이 있게 종료되면 챗봇의 요약이 아닌, **직접 타이핑한 답변 원본만을** 복사하여 `raw_store.yaml`에 붙여넣는다.

## Step-by-Step 시작하기

**1. `raw_store.yaml` 파일 세팅**
안전한 로컬 환경에 `raw_store.yaml` 파일을 생성한다. 
```yaml
entries: []
```
> **주의**
> 사용자의 답변은 민감한 사적 데이터이므로, 이 저장소를 클론받은 경우 절대 원본 데이터 파일을 커밋하지 않는다.

**2. 메서드 선택 및 수행**
원하는 기법을 선택하여 안내에 따라 수행한다.

| Method | 특성 | 참고 문서 |
|---|---|---|
| CCRT | 정적 (혼자 생각하고 기록) | **[핵심 갈등 도식 (CCRT)](<핵심 갈등 도식 (CCRT).md>)** |
| Feared Self | 정적 (혼자 생각하고 기록) | **[두려운 자기와 조기 경보](<두려운 자기와 조기 경보.md>)** |
| Triadic Elicitation | 정적 + 수치 연산 (파이썬 코드 필요) | **[삼항 도출](<삼항 도출.md>)** |
| Laddering | 동적 (LLM 챗봇 꼬리질문 필요) | **[래더링](<래더링.md>)** |

**3. 최종 분석 (데이터 누적 후)**
충분한 데이터가 모였다면, **모든 추출이 끝난 후에만 LLM을 활용하여 분석한다.** 새로운 챗봇 창을 열어 `raw_store.yaml`의 전체 내용을 복사하고, 패턴 분석을 요청한다.

## 작업 예시: 핵심 갈등 도식 (CCRT) 세션

**추출 질문 읽기:**
최근 6개월 이내의 실제 관계 장면을 떠올리고, 본인의 소망, 상대의 실제 반응, 본인의 대응을 타이핑한다.

**원본 데이터를 `raw_store.yaml`에 추가 (사전 해석 금지):**

```yaml
entries:
  - id: ccrt0001
    ts: 2026-06-20T14:00:00+09:00
    method: ccrt
    object_tag: "동료"           # 실명 대신 관계 유형 태그 사용
    raw_wish: "내가 힘든 걸 굳이 말 안 해도 알아봐 주길"
    raw_response_other: "전혀 눈치 못 채고 평소처럼 일 얘기만 함"
    raw_response_self: "혼자 서운해하다 더 무뚝뚝하게 굴어서 사이가 멀어짐"
```

**금지된 작성 방식 (사전 해석이 주입된 경우):**
```yaml
# 잘못된 예: 실제 발화가 아닌 요약된 해석
raw_wish: "인정 욕구가 강함, 의존적 성향"
```
**핵심 원칙:** 데이터 필드에는 해석이나 분석이 아닌 **"사용자가 실제로 한 생각과 말"만이** 들어가야 한다.

---

## 선택 사항: 수동 검증 테스트

LLM이 사용자의 심리 패턴을 제대로 파악했는지 검증하는 테스트 절차이다. 순서를 철저히 지켜야 한다.

1. **새 창에서 질문:** 데이터 전체를 챗봇에 붙여넣고 새로운 CCRT 에피소드에서의 본인 반응을 구체적으로 예측해 보라고 지시한다.
2. **독립적 작성:** LLM의 예측 결과를 확인하기 전에, 반드시 본인의 실제 대답을 먼저 기록해 둔다.
3. **비교 채점:** 본인이 기록한 대답과 LLM이 예측한 텍스트 간의 논리적 일치도를 검토한다.

이 절차는 본 프로젝트의 방법론이 환각이나 바넘 효과에 기대지 않음을 확인하는 **[검증 모드](<검증 모드.md>)**의 핵심 구조이다.

## 데이터 필드 네이밍 규칙

| Field prefix | Meaning |
|---|---|
| `raw_*` | 사용자의 실제 답변 원문 (의역 절대 금지) |
| `id` | 메서드 접두사와 일련번호 (예: `ccrt0001`) |
| `ts` | ISO 8601 형식의 타임스탬프 |
| `method` | 기법의 식별자 |
| `object_tag` | **관계 유형 태그 (예: "동료").**<br> *답변에 실명이 포함되었다면 저장 전 반드시 태그로 치환하여 가명화한다.* |
