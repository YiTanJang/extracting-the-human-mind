# Methodology: 맥락 기반 제로섬 가치 할당 파이프라인 (Context-Based Zero-Sum Value Allocation)

> **한 줄 정체성:** Schwartz 가치 인플레이션 통제 + Tetlock 전환 조건. 맥락별 긴장 상황에서 사용자가 *직접 적은* 5~6개 가치에 심리 코인 100개를 제로섬 배분시키고, 1위↔최하위가 뒤집히는 전환 조건(Flip Condition)을 캔다(고정 메뉴 없는 개방형).

## 0. 이론적 전제 및 설계 원칙

* **가치 인플레이션 통제 (Controlling Value Inflation):** 일반적인 리커트 척도(1~7점)로 중요도를 물으면 대부분의 가치에 만점을 주는 천장 효과(Ceiling effect)가 발생한다. 우선순위는 오직 제한된 자원(시간/에너지/돈)을 나눌 때만 드러난다. (Schwartz의 보편적 가치 이론)
    
* **맥락 종속성 (Contextual Dependency):** 가치의 위계는 고정되어 있지 않다. 업무 맥락에서의 1위 가치(예: 효율성)와 관계 맥락에서의 1위 가치(예: 수용성)는 다르게 작동하므로, 반드시 구체적인 씬(Scene)을 설정한 뒤 할당하게 해야 한다.
    
* **통합적 복잡성과 예외 (Integrative Complexity, Tetlock):** 절대적인 가치는 없다. 상위 가치를 포기하고 하위 가치를 선택하게 만드는 '전환 조건(Boundary/Flip Condition)'을 알아야, 챗봇이 모순된 행동을 예측하고 시뮬레이션할 수 있다.

- **개방형 자기 도출 (Open Self-Elicitation, §C-4 결정):** 사전 정의된 가치 목록(메뉴)을 제시하지 않는다. 메뉴를 주는 순간 그건 설계자 프레임 주입(무오염 위반)이고, 그렇다고 이전 추출(삼항·래더링)에서 자동 시딩하면 자기-편향 증폭이 된다. 따라서 사용자가 *이 맥락에서 자기에게 중요한 것*을 직접 빈칸에 적게 한 뒤, 그 자기 언어 위에서 배분하게 한다. 이 시스템은 인구 비교(nomothetic)가 아니라 개인 특이적(idiographic)이므로, 비교 가능한 표준 목록을 포기해도 손해가 없다 — 오히려 무오염 원칙과 더 일관된다.
    

## 단계 1. 개방형 가치 자기 도출 (Open Value Self-Elicitation)

고정 메뉴 대신, 사용자가 *이 맥락에서 자기에게 중요한 것*을 직접 빈칸에 적는다(§0 개방형 자기 도출). 다른 모든 개방형 방법과 동일하게, 설계자 옵션도·이전 raw 자동 시딩도 없다.

### 1-1. 맥락 닻 + 자유 기입

* **프롬프트 (UI):** *(해당 도메인의 긴장 상황을 먼저 제시 — 단계 2-1)* `"지금 이 상황에서, 당신이 지키거나 얻고 싶은 것을 5~6개, 당신의 말로 적어주세요. (정답은 없습니다. 떠오르는 대로요.)"`
* **동작:** 사용자가 5~6개의 가치 라벨을 자유 기입한다($Raw\_Values$). 단어·짧은 구 모두 허용.

### 1-2. 기입 위생 (Anti-Fatigue, Anti-Generic)

* **개수 가이드:** 너무 적으면(≤2) 제로섬이 무의미하므로 최소 4개를 부드럽게 유도하되 강제하지 않는다.
* **일반어 재유도(1회, 비강제):** "성공", "행복" 같은 지나치게 추상적·바넘적 단어만 나오면 *교정하지 않고* "그게 당신에게 구체적으로 어떤 모습인지 한 단어 더 붙여볼래요?"를 1회 권한다. 끝까지 추상어를 고집하면 그대로 수용·저장(그 자체가 신호 — §5).
* **무오염:** 예시 단어를 화면에 띄우지 않는다(예시가 곧 메뉴가 된다).

## 단계 2. 맥락 부여 및 제로섬 할당 (Contextual Zero-Sum UI)

단순히 "얼마나 중요합니까?"라고 묻는 대신, 압박감이 있는 특정 상황을 부여하고 심리 코인을 분배하게 한다.

### 2-1. 맥락 세팅 (UI 프롬프트)

사용자가 현재 진입한 도메인에 맞춰 긴장감 있는 딜레마 상황을 제시한다.

* **업무(Work) 맥락:** `"지금 당신은 한정된 시간과 예산 속에서, 성패가 불확실한 중요 프로젝트의 책임을 맡았습니다."`
    
* **관계(Relation) 맥락:** `"지금 당신은 서로 입장이 완전히 다른 두 사람 사이에서 갈등을 중재해야 하는 피곤한 상황에 놓였습니다."`
    

### 2-2. 강제 할당 메커니즘 (Forced Distribution)

* 사용자에게 100개의 심리 코인(Points)을 지급한다.
    
* 단계 1에서 *본인이 직접 적은* 5~6개의 가치 위에서, 사용자는 각 가치에 코인을 분배한다.
    
* **동작 (제약 조건):** 할당된 코인의 합이 정확히 **100점**이 되어야만 '다음' 버튼이 활성화된다. (하나에 100을 다 주거나, N빵을 하는 등 극단적 선택도 모두 허용하되, 합은 무조건 100이어야 함)
    

## 단계 3. 전환 조건 추출 (Boundary & Flip Condition Elicitation)

할당이 끝나면 알고리즘이 즉시 1위(가장 코인을 많이 준 가치)와 최하위(가장 적게 준 가치)를 식별하여, 가치 위계가 뒤집히는 '예외 상황'을 캐묻는다.

* **프롬프트 (UI):**
    
  `"당신은 이 상황에서 [ 1위 가치(예: 내 방식대로 통제함) ]에 가장 높은 비중을 두었고, [ 최하위 가치(예: 팀원들과의 화합) ]은 과감히 후순위로 미뤘습니다.`
    
  `그렇다면, 어떤 예외적인 상황이나 조건이 발생하면 당신은 1위인 [내 방식대로 통제함]을 포기하고 기꺼이 꼴찌였던 [팀원들과의 화합]을 선택하시겠습니까?"`
    
* **동작:** 사용자가 텍스트를 입력한다. ($Raw\_Flip\_Condition$)
    
* **의의:** 100점 만점에 10점을 준 가치라도 버려진 것이 아니다. 특정 조건이 만족되면 언제든 1위로 튀어 오를 수 있는 트리거를 확보하는 것이다. (Argyris의 Theory-in-use 행동 법칙 발견)
    

## 단계 4. 산출물 및 데이터 저장 (Raw Store Append)

할당된 수치와 전환 조건은 계산이나 요약 없이, 당사자의 우선순위 위계 그대로 YAML에 저장된다.

YAML

```
  - id: v0001
    ts: 2026-06-19T22:30:00+09:00
    method: value_allocation
    context: work_crisis_scenario       # 맥락 명시
    allocations:                        # 라벨은 전부 사용자가 직접 적은 verbatim (메뉴 아님, construct_id 없음 — §C-1/C-4)
      - {raw_verbatim: "내 방식대로 통제함", points: 45}
      - {raw_verbatim: "확실한 성과와 보상", points: 30}
      - {raw_verbatim: "예측 가능한 안정성", points: 15}
      - {raw_verbatim: "타인에게 민폐 안 끼침", points: 10}
      - {raw_verbatim: "팀원들과의 화합", points: 0}
    flip_condition:                     # 1위(통제)를 버리고 최하위(화합)를 택하는 예외 조건
      top_value: "내 방식대로 통제함"
      bottom_value: "팀원들과의 화합"
      raw_condition: "내 방식대로 밀어붙이다가 누군가 신체적으로 번아웃이 와서 쓰러지기 직전일 때"
```

## 단계 5. 알려진 한계 및 활용 원칙 (Limitations)

* **자기 도출의 트레이드오프 (Open-Elicitation Cost):** 메뉴를 없애 설계자 프레이밍은 제거했지만, 대신 두 위험을 떠안는다 — (1) 사용자가 사회적으로 바람직한 추상어("성장", "배려")만 떠올려 적을 수 있고, (2) 자유 기입이라 인구 간 비교가 불가능하다. (1)은 단계 1-2의 비강제 재유도로 일부 완화하되 끝까지 추상어를 고집하면 *그 자체를 신호*로 저장한다(방어기제 강도). (2)는 본 시스템이 개인 특이적이라 애초에 비교를 목표하지 않으므로 손실이 아니다.
    
* **Part 2 활용 원칙 (모순 시뮬레이션):** LLM은 이 데이터를 "이 사람은 자율성을 45%만큼 중요하게 생각하는 사람"이라는 정적인 성격 파라미터로 쓰면 안 된다. 시나리오를 시뮬레이션할 때, "평소에는 \[자율성\]을 추구하며 독단적으로 행동하겠지만, 만약 '윤리적 비난'이라는 Flip Condition이 감지되면 즉시 \[사회적 기여/팀의 안위\]를 챙기는 방향으로 태세를 전환할 것이다"라는 동적 분기점(If-Then Rule)으로 활용해야 한다.
